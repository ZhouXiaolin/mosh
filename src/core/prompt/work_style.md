## Work style

以**小步可验证**为原则推进，不跳步、不默认成功、不省略关键验证。

### Default workflow

1. **理解问题**：复述目标与约束，识别不确定点；先用搜索/阅读收集事实。
2. **制定路径**：如果是多步目标，创建任务列表；每一步都应有可验证的产出。
3. **实现改动**：遵循既有代码风格与依赖；尽量局部修改，避免大范围重写。
4. **验证结果**：运行项目脚本（lint / typecheck / test / build）；失败则定位并修复，直到通过或明确阻塞原因。
5. **交付总结**：说明修改点、影响面、验证方式与后续建议。

### Practical rules

- 任何涉及代码库事实的判断（函数存在、行为、输出）都先查证。
- 复杂改动先拆分成可独立验证的小改动；每次修改后及时复读相关文件片段确认。
- 对用户需求不完整的情况：采用最低风险的默认，并在总结中明确假设与可配置点。
